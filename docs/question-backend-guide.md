# Question コンテキスト ドメイン知識

Question（問い）のビジネスルールと設計判断。
横断的なアーキテクチャルールは `docs/backend-architecture-guide.md` を参照。

---

## ドメイン概念

### Question（問い）

ユーザーが設定する「問い」の親レコード。発酵プロセス（AI 分析）の軸となる。

- アーカイブ可能（非表示にするが削除はしない）
- Oryzae（AI）が新しい問いを提案できる。ユーザーの承認が必要
- **アクティブな問い** = 非アーカイブ かつ ユーザー承認済み

### QuestionTransaction（問いトランザクション）

問いの文面の版管理。**append-only**（更新ではなく追記）。

- 文面は最大 64 文字
- バージョン番号は question_id ごとに単調増加（1始まり）
- Oryzae が文面の更新を提案できる。ユーザーの承認が必要
- **最新の有効な文面** = 同一 question_id の承認済み transaction のうち最大バージョン

### EntryQuestionLink

Entry と Question の N:M 紐付け。ドメインモデル化しない純粋な関連テーブル。

## ビジネスルール

### 問い管理
- ユーザーあたり **最大3つ** のアクティブな問い
- ユーザーは任意のタイミングで問いを作成・編集・アーカイブできる
- アーカイブした問いは復活可能（ただしアクティブ問い上限を超えない）
- 問いの文面を編集すると、新しい transaction が追記される（過去の文面は履歴として残る）

### Oryzae 提案フロー

発酵プロセスの結果として、Oryzae が2種類の提案を行う:

1. **新しい問いの提案**: Question 自体が未承認状態で作られる
2. **既存の問いの更新提案**: 既に承認済みの Question に対し、未承認の Transaction が追記される

ユーザーの対応:
- **承認**: 問い/トランザクションが有効化される
- **却下**: 新規問い提案なら Question ごと削除。更新提案なら該当 Transaction のみ削除

### なぜ append-only か

問いの変遷履歴を完全に保持することで、「問いがどう育ったか」のタイムラインを可視化できる。これは Oryzae の理念（問いを育てる）の中核機能。

## 設計判断の記録（ADR）

### ADR-001: Question と QuestionTransaction の分離

**決定**: 問いの「存在」（Question）と「文面」（QuestionTransaction）を別テーブルで管理する。

**理由**: 文面の版管理を append-only で行うため。問い自体のメタデータ（アーカイブ、承認状態）と文面の履歴は異なるライフサイクルを持つ。

### ADR-002: アクティブ問い上限 3

**決定**: ユーザーあたりのアクティブ問いを最大3に制限する。

**理由**: 問いの数を絞ることで、各問いへの注意力を集中させる。Oryzae のプロダクト哲学（「問いを育てる」）に基づく意図的な制約。

**詳細**: `docs/archive/question_proposal.md` に初期要件を記録。
